# 项目实施文档索引

这组文档是 LOL 海克斯助手正式版的需求、架构、验收和实施依据。

旧 POC 仓库只作为技术验证和证据留存来源，不再继续叠加完整产品功能。本项目从这些文档出发，建立清晰的需求、架构、验收标准和工程边界。

## 文档列表

- `01-product-requirements.md`：产品需求、用户流程、MVP 范围和非目标。
- `02-architecture-and-tech-selection.md`：架构设计、模块划分、数据流和技术选型。
- `03-acceptance-and-risks.md`：阶段验收、日志样本要求、风险和禁止事项。
- `04-implementation-notes.md`：从 POC 迁移到本项目时的实施注意事项。

## 旧 POC 中已经验证的关键结论

- 显示器级截图可用于 LOL 局内无边框模式；独占全屏存在旧帧和不刷新风险，不作为 MVP 默认前提。
- OCR 核心路线应使用本地离线 ONNX 推理。旧 POC 已验证 PP-OCRv4 rec 模型经 Rust `ort` crate 在 5 张真实样本上 15/15 通过。
- Overlay 应采用透明、置顶、点击穿透窗口，不使用注入或 Hook。
- 英雄和等级只走 Riot / LOL 本地 Live Client Data API，不通过进程扫描、窗口标题或读内存判断。
- ApexLOL 查询必须保留来源和失败兜底，不能伪造无来源数据。

## 本项目不可突破的边界

- 不注入游戏进程。
- 不 Hook 游戏渲染或输入。
- 不读游戏内存。
- 不自动点击。
- 不自动替玩家选择海克斯。
- 不为了识别结果添加样本特化 OCR 错字表。
- 不把失败的 ApexLOL 查询伪装成真实推荐。
