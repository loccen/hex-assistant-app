# 性能优化待办

本文档只记录运行时识别与 Overlay 链路的性能优化现状和后续待办，不包含 UI 调整方向。

## 当前结论

- 识别到 Overlay 的主观延迟已经从此前的秒级等待明显下降到百毫秒级处理加轮询等待。
- 当前版本已完成：
  - 运行时 OCR 与 ONNX Runtime 复用，避免每轮重新初始化。
  - 运行时截图改为内存链路，默认不再每轮落盘 PNG / JSON。
  - 增加截图、区域分析、OCR 冷启动 / 复用命中、诊断落盘等关键阶段日志。
  - 轮询改成分层策略：空闲低频、海克斯阶段高频。

## 最新观察

基于最近一次有效局内日志分析：

- `active` 模式当前轮询间隔是 `1000ms`。
- 单轮活跃处理成本大致在 `90ms` 到 `110ms`。
- 运行时截图通常在 `30ms` 到 `42ms`，首轮偶发更高。
- 区域分析通常在 `0ms` 到 `1ms`。
- Overlay 更新仍然是毫秒级，不是瓶颈。

因此，剩余体感延迟的主要来源已经不再是截图落盘、OCR 初始化或 Overlay 渲染，而是运行时控制流本身。

## 后续优化顺序

### 1. Live Client 读取链路

优先级最高。

当前问题：

- 每轮都会先执行 `fetch_resolved_player_snapshot()`。
- 该路径每次都会新建阻塞式 `reqwest` client。
- 请求是串行的，至少会打 `activeplayername` 和 `allgamedata`，失败时还可能追加 `playerlist`。
- 失败态没有明显断路器，会持续空转重试。

建议方向：

- 复用 HTTP client。
- 缩减请求次数，优先评估是否能只依赖更少的接口组合。
- 为连续失败场景增加断路器或冷却机制。

### 2. Overlay Hide 去重

优先级高。

当前问题：

- 在 `alreadyHidden`、`noWindow` 这类状态下，仍然会重复进入 hide 流程。
- 失败态日志中大量轮次都在重复执行同样的 hide 决策与同步。

建议方向：

- 为运行时 overlay plan 增加去重。
- 当目标状态和当前状态一致时，直接跳过 overlay 调用。
- 减少 `alreadyHidden` 和等价状态下的重复日志。

### 3. 失败态 / 无变化态轮询退避

优先级高。

当前问题：

- `LiveClientUnavailable`、`InvalidLiveClientData` 等暂停态仍会固定频率轮询。
- 即使没有新数据、overlay 已隐藏，也会继续周期性重复完整决策。

建议方向：

- 对失败次数做累计。
- 对连续失败场景采用更长退避，必要时采用分段或指数退避。
- 对状态未变化轮次应用更保守的轮询间隔。

### 4. 运行时日志降采样

优先级中高。

当前问题：

- 一次轮询会写多条 `app.jsonl`。
- 失败态还会叠加 overlay debug log。
- 当前结构化日志是每次打开文件并写入，没有进程内缓冲。

建议方向：

- 对重复的运行时事件做降采样或合并。
- 对失败态日志按固定窗口聚合。
- 对 overlay debug log 区分普通模式与诊断模式。

### 5. 活跃轮询间隔继续下探

优先级中。

当前问题：

- 现在 `active` 轮询仍是 `1000ms`。
- 单轮活跃处理成本已经接近 `100ms`，还有继续缩短间隔的空间。

建议方向：

- 先尝试把活跃轮询从 `1000ms` 下调到 `500ms`。
- 需要配合观察 CPU、截图稳定性和 live client 请求压力。

### 6. 设置 / 词库重复读取

优先级中。

当前问题：

- 运行时检测路径仍会重复读取设置。
- OCR 路径仍会重复加载词库摘要信息。

建议方向：

- 将设置和词库读取改为会话内缓存。
- 保留手动刷新或失效策略，避免长期脏读。

### 7. 区域分析与其余细枝末节

优先级低。

当前结论：

- 区域分析已经很轻，不是当前瓶颈。
- Overlay 渲染也不是瓶颈。
- 运行时截图落盘问题已经基本解决。

只有在前面几项完成后，才需要继续抠这些细节。

## 暂不纳入本文档

- 主界面与校准向导 UI 调整。
- Overlay 视觉风格优化。
- 文案、布局和交互体验问题。

这些内容由后续单独需求驱动，不在本性能待办中展开。
