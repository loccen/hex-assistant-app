# 本项目验收与注意事项

## 1. 文档定位

本文档面向 LOL 海克斯助手本项目实施，不沿用 POC 的最终验收结论。POC 已验证“显示器级截图 + 用户校准 + OCR + Overlay + Live Client 状态机 + ApexLOL 查询”的技术路线，但本项目仍必须按本文重新验收环境、真实游戏样本、打包发布和安全边界。

本轮起草依据：

- `README.md`
- `lol-hex-assistant-acceptance.md`
- `artifacts/ocr-race/results/overall-report.md`
- `artifacts/ocr-race/results/rapidocr-rust-regression/report.md`

旧 POC 中已经观察到的结论只能作为风险输入，不能直接写成本项目已通过。尤其是独占全屏旧帧、不同地图和 UI 缩放、Overlay 点击穿透、OCR 模型和 ORT 动态库打包、ApexLOL 页面变化，都必须在本项目中重新留样验收。

## 2. 总体验收边界

本项目只做信息展示和人工决策辅助。所有识别和展示都必须来自显示器级截图、用户校准区域、低频 OCR、Live Client 本地允许接口、本地缓存和 ApexLOL 页面解析结果。

### 2.1 明确禁止事项

- 禁止注入游戏进程、DLL 注入、驱动级能力或任何 Hook。
- 禁止读取、扫描或修改游戏内存。
- 禁止自动点击、自动选择、自动重随、自动确认或模拟玩家输入。
- 禁止高频截图或高频 OCR；OCR 只能在选择阶段按低频节奏处理校准区域。
- 禁止为了通过样本验收编写样本特化 OCR 补丁、特化错字表或低置信度强行匹配。
- 禁止伪造 ApexLOL 数据，禁止把其他英雄、其他海克斯或无来源数据展示成 ApexLOL 结果。
- 禁止用“非黑屏”单独证明截图通过；疑似旧帧必须按失败或风险项处理。

### 2.2 验收记录通用要求

每个验收项都必须保留：

- 输入：测试环境、游戏显示模式、地图、分辨率、UI 缩放、样本路径或账号状态。
- 动作：人工操作步骤，包含点击入口、切换窗口、校准、查询或打包命令。
- 期望结果：页面表现、日志状态、错误提示和不能发生的行为。
- 留存：日志、JSON、PNG、录屏、打包产物、缓存样本或测试记录路径。

同一关键场景建议连续复测 3 次。结果不一致时，不得写成通过，必须记录差异和下一步定位项。

## 3. 分阶段验收

### 3.1 环境 / 安装

目标：确认本项目能在受控运行时下安装、开发运行和构建，不依赖开发者机器上的偶然全局环境。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 本项目代码仓库；`mise.toml` 或等效项目运行时配置；Windows 桌面环境和 WSL Ubuntu 开发环境；Node.js、Rust、Tauri 所需系统依赖。 |
| 动作 | 先检查项目运行时配置；执行 `mise trust`、`mise install`、`mise exec -- npm install`；执行 `mise exec -- npm run build`；在 Windows 桌面环境执行 `mise exec -- npm run tauri dev`。 |
| 期望结果 | 依赖安装不依赖全局 `node`、全局 `rust` 或其他运行时；前端构建成功；Tauri 开发窗口能启动；缺失系统依赖时有明确错误说明和补充记录。 |
| 留存 | `mise.toml`、安装日志、构建日志、Tauri 启动日志、依赖锁文件、测试日期和机器信息。 |

注意事项：

- 修改运行时版本必须同步更本项目级配置，不能只在本机临时切换。
- 如果只修改文档，可以不运行构建，但提交前仍必须执行 `git diff --check`。
- 真实截图、Overlay 和点击穿透不能只在 WSL 中验收，必须进入 Windows 桌面环境。

### 3.2 截图 / 校准

目标：确认本项目能获取可复查截图样本，并通过用户校准得到名称 OCR 区域、卡片底部锚点和展示 / 隐藏按钮区域。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 至少覆盖窗口、无边框、独占全屏三种显示模式；至少覆盖嚎哭深渊、莲华栈桥、屠夫之桥或本项目实际支持地图；覆盖不同分辨率、画质和 UI 缩放。 |
| 动作 | 选择目标显示器；填写显示模式和地图备注；运行截图诊断；独占全屏使用 8 秒或 12 秒延迟后切回游戏；在可见三张海克斯卡片的样本上框选 3 个名称区域、3 个底部锚点和 1 个底部按钮区域；保存并重新加载校准配置。 |
| 期望结果 | PNG 不是黑屏；截图尺寸、亮度统计和耗时被记录；旧帧检测没有命中历史成功样本；校准回显框准确落在目标区域；重新加载后坐标不漂移；分辨率、显示器或 UI 缩放变化后能提示重新校准。 |
| 留存 | `diagnostic.log`、`diagnostic.json`、原始 PNG、校准 JSON、校准回显图、旧帧匹配记录、每次测试的显示模式和地图备注。 |

风险与处理：

- POC 已发现独占全屏可能拿到旧帧；本项目默认验收应优先使用无边框，独占全屏只在连续刷新样本证明可用后才可列为支持。
- 不同地图、画质、分辨率和 UI 缩放必须分别记录，不能用一张成功样本推断全部通过。
- 多显示器环境必须记录目标显示器编号、尺寸、缩放和主副屏关系。

### 3.3 OCR

目标：确认 OCR 只处理用户校准的三个名称区域，能输出可复盘的裁剪图、候选文本、置信度和词库匹配结果。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 完成校准的真实无边框样本；不同地图、画质、UI 缩放和遮挡状态的样本；OCR 模型文件和运行时库；标准海克斯词库。 |
| 动作 | 使用真实截图离线回放 OCR；在选择阶段展开三张卡片后触发低频 OCR；分别测试正常识别、词库纠错、低置信度、手动修正、单 slot 重随变化。 |
| 期望结果 | 每个 slot 输出 raw、focused、tight-line、enhanced 等裁剪或增强图；记录原始文本、置信度、匹配分和最终名称；低置信度不直接查询 ApexLOL；单 slot 重随后只刷新该 slot；面板收起时停止 OCR；详情浮层遮挡底部按钮但卡片仍展开时不停止 OCR。 |
| 留存 | OCR 回放报告、三槽裁剪图、增强图、候选文本日志、词库匹配日志、低置信度样本、手动修正记录、重随前后 slot 变化日志。 |

风险与处理：

- OCR 模型路径必须同时覆盖开发环境和打包环境；生产包不能依赖开发机上的 pip 包路径。
- ORT 动态库路径必须明确；Windows 打包需要验证 `onnxruntime.dll` 随包可用。
- 不允许用样本特化补丁制造 100% 通过。任何纠错规则都必须来自通用词库匹配规则，并在新样本上回归。
- 旧 POC 已有 OCR race 总报告和 Rust ORT 回归报告；本项目实施前应重新生成等价报告，不能直接把 POC 报告当作本项目通过结论。

### 3.4 Overlay

目标：确认 Overlay 按校准锚点显示三张说明卡片，并且不影响玩家操作。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 完成校准的真实游戏环境；窗口、无边框和计划支持的全屏场景；单显示器和多显示器环境；成功、无数据、请求失败三类展示内容。 |
| 动作 | 打开海克斯选择面板；观察 Overlay 三张卡片位置；点击三张海克斯卡片、重随按钮、展示 / 隐藏按钮和游戏热键；收起并重新展开面板；切换目标显示器后复测。 |
| 期望结果 | 三张说明卡片位于对应海克斯卡片下方，不遮挡名称、选择入口、重随按钮和底部按钮；Overlay 不抢焦点、不拦截点击、不影响游戏热键；面板收起时隐藏，展开时恢复；校准缺失或越界时不显示并提示重新校准。 |
| 留存 | Overlay 展示截图或录屏、隐藏截图或录屏、点击穿透测试记录、目标显示器日志、三张卡片坐标和尺寸日志、异常提示样本。 |

风险与处理：

- 点击穿透必须在真实 Windows 桌面环境验收，不能只看开发窗口。
- 多显示器下要验证 Overlay 绑定正确显示器，不得出现在副屏或错位。
- Overlay 异常时只能隐藏并提示，不能自动点击、自动展开或自动修正游戏状态。

### 3.5 Live Client 状态机

目标：确认状态机只使用 Live Client 本地允许接口和截图 / OCR 状态，正确处理等级、面板收起、展开、多档位和重随。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 可进入游戏的测试账号或录制样本；等级低于 3、达到 3 / 7 / 11 / 15、多档位积累、面板收起、面板展开、重随后 slot 变化等状态。 |
| 动作 | 启动助手监听 Live Client；记录英雄和等级变化；达到海克斯等级后观察待处理队列；手动展开 / 收起面板；手动重随某个 slot；制造 Live Client 暂时不可用或 OCR 失败。 |
| 期望结果 | 等级低于 3 时不 OCR、不展示 Overlay；达到 3 / 7 / 11 / 15 时生成待处理队列；按钮存在但卡片不可见时进入面板收起，不标记完成；卡片展开时允许 OCR 和展示；多档位逐轮处理；重随后只刷新变化 slot；异常时提示并等待人工操作。 |
| 留存 | 状态迁移日志、Live Client 返回摘要、英雄和等级日志、待处理队列日志、面板收起 / 展开样本、重随 slot 变化日志、异常恢复日志。 |

风险与处理：

- 卡片收起不等于选择完成；不能因为按钮区域被遮挡就结束当前轮。
- 按钮区域只能作为辅助状态信号；卡片可见性优先。
- Live Client 日志不得记录敏感令牌，也不得引入读取游戏内存的实现。

### 3.6 ApexLOL 查询

目标：确认助手能用当前英雄和标准海克斯名称查询 ApexLOL，缓存并展示有来源的短内容。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 当前英雄名称；OCR 词库匹配后的标准海克斯名称；网络正常、断网、ApexLOL 页面结构变化、无数据等场景。 |
| 动作 | 对三张海克斯分别发起查询；重复查询同一英雄 + 海克斯组合；断网后查询；准备无法解析或无数据页面；重随某个 slot 后重新查询该 slot。 |
| 期望结果 | 查询粒度为“英雄 + 海克斯”；缓存命中有日志；成功结果包含评级、短说明、提醒、来源 URL 和获取时间；失败或解析不到可靠内容时显示“暂无数据”；不同英雄同名海克斯不串数据；重随后只查询变化 slot。 |
| 留存 | 请求 URL、请求耗时、缓存命中日志、解析状态、错误信息、缓存记录、成功 / 缓存 / 失败 / 无数据四类展示样本。 |

风险与处理：

- ApexLOL 页面结构可能变化，解析失败必须可见、可记录、可回退为“暂无数据”。
- 禁止伪造 ApexLOL 数据，禁止把请求失败写成成功。
- 展示层必须标明“数据来源：ApexLOL”，来源入口不得拦截游戏点击。

### 3.7 端到端一局实测

目标：确认从安装、校准、进入游戏、识别、查询、展示、重随到结束一局的完整流程可用。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 一局真实游戏；优先无边框模式；已完成校准；网络可访问 ApexLOL；启用日志和样本导出。 |
| 动作 | 启动助手；进入游戏；在 3 / 7 / 11 / 15 级选择阶段手动展开面板；观察 OCR、ApexLOL 查询和 Overlay；手动收起 / 展开面板；至少执行一次重随；局后导出完整日志和样本。 |
| 期望结果 | 助手不影响玩家操作；每个选择阶段状态机迁移清晰；三张卡片能识别或给出低置信度提示；Overlay 展示短内容并可隐藏；重随只刷新变化 slot；失败时不伪造结果，不自动操作。 |
| 留存 | 完整运行日志、状态机日志、OCR 样本、Overlay 截图或录屏、ApexLOL 查询日志、缓存记录、局内问题记录、测试人和测试时间。 |

风险与处理：

- 如果一局中出现截图黑屏、旧帧、Overlay 错位、点击穿透失败、OCR 低置信度或 ApexLOL 失败，必须单独记录，不得合并成“端到端通过”。
- 端到端验收至少要覆盖一局正常完成和一局包含异常兜底的流程。

### 3.8 打包发布

目标：确认发布包在干净 Windows 环境中可安装、可启动、可 OCR、可查询，并包含必要模型和动态库。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 干净 Windows 测试机或干净用户目录；正式打包产物；OCR 模型；ORT 动态库；默认配置和空缓存。 |
| 动作 | 执行发布构建；在干净环境安装；启动应用；运行截图诊断；导入或重新校准；执行 OCR 回放；执行 ApexLOL 查询；卸载后检查残留。 |
| 期望结果 | 安装和启动成功；OCR 模型路径正确；`onnxruntime.dll` 或等价动态库可加载；缺少资源时有明确错误提示；缓存和配置写入用户数据目录；卸载不破坏用户手动导出的样本。 |
| 留存 | 构建日志、安装包路径和校验值、安装日志、启动日志、OCR 加载日志、模型和动态库清单、干净环境测试记录。 |

风险与处理：

- ORT 动态库和 OCR 模型是打包高风险项，必须在没有开发环境变量的机器上验证。
- 不能依赖开发机上的 `ORT_DYLIB_PATH`、pip 包目录或源码相对路径。
- 发布说明必须写明推荐无边框模式、已知独占全屏限制和日志导出位置。

### 3.9 安全边界

目标：确认本项目没有越过低可疑面边界，不改变游戏、不替玩家操作。

| 项目 | 验收要求 |
| --- | --- |
| 输入 | 完整源码、依赖清单、运行日志、打包产物、用户操作流程。 |
| 动作 | 搜索和审查注入、Hook、内存读取、模拟输入、自动选择、高频截图 / OCR 相关实现；运行端到端场景并观察用户操作边界；检查日志和发布说明。 |
| 期望结果 | 源码和依赖中没有注入、Hook、读内存、自动点击、自动选择；运行时不会抢焦点或模拟输入；异常兜底只提示用户或允许手动修正；日志不包含敏感令牌。 |
| 留存 | 代码审查记录、关键字搜索记录、依赖审查记录、运行日志、发布说明、安全边界确认记录。 |

建议关键字包括但不限于：`inject`、`hook`、`memory`、`ReadProcessMemory`、`SendInput`、`mouse_event`、`click`、`auto select`、`ocr interval`、`capture interval`。

## 4. 已知风险总表

| 风险 | 影响 | 本项目验收要求 |
| --- | --- | --- |
| 独占全屏旧帧 | 可能展示过期画面，导致 OCR 和 Overlay 结论错误 | 必须用连续截图和旧帧哈希检测证明可用；否则默认建议无边框。 |
| 不同地图 / 画质 / UI 缩放 | 卡片位置、字体清晰度和按钮区域可能变化 | 必须按地图、画质、缩放分别留样，失败场景单独记录。 |
| Overlay 点击穿透 | 可能阻挡玩家选择或影响热键 | 必须在真实 Windows 桌面、多显示器和目标显示模式下实测。 |
| 多显示器 | 可能截图或 Overlay 绑定错误屏幕 | 日志必须记录目标显示器，测试主屏和至少一个非主屏。 |
| ORT 动态库打包 | 发布包可能启动成功但 OCR 加载失败 | 必须在干净 Windows 环境验证动态库随包可加载。 |
| OCR 模型路径 | 开发环境可用，生产包找不到模型 | 必须验证模型被打入资源目录，并记录加载路径。 |
| ApexLOL 页面变化 | 查询失败、解析字段缺失或展示旧结构 | 必须保留解析失败兜底和来源 URL，失败显示“暂无数据”。 |
