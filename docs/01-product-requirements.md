# LOL 海克斯助手产品需求文档

## 1. 文档定位

本文档面向本项目的正式实现，不复述旧 POC 的阶段过程。旧 POC 已验证显示器级截图、用户校准、本地离线 OCR、透明 Overlay、LOL Live Client Data API 和 ApexLOL 查询等路线可继续推进；本项目基于这些结论重新整理产品边界、用户体验和验收口径，避免继续在旧 POC 仓库中堆叠临时代码。

本项目的核心定位是：在 LOL 海克斯乱斗局内，为玩家展示当前英雄与三个海克斯符文的联动信息，帮助玩家更快做人工判断。助手只提供信息提示，不替玩家操作游戏。

## 2. 产品目标

- 在玩家进入海克斯选择阶段时，自动识别当前英雄、等级和三张海克斯符文名称。
- 在三张海克斯卡片下方展示简短、可读、可快速比较的推荐信息。
- 明确展示数据来源，并在请求失败或解析失败时给出“暂无数据”等可理解状态。
- 保持低可疑面：不进入游戏进程、不干预玩家输入、不影响游戏公平性。
- 默认支持无边框模式下的局内体验；独占全屏只作为风险项和后续研究项。
- 提供可复查的校准、识别、查询和异常记录，方便定位截图、OCR、数据来源或 Overlay 问题。

## 3. 目标用户与场景

### 3.1 目标用户

- 经常游玩 LOL 海克斯乱斗，希望快速判断海克斯强弱和联动的玩家。
- 熟悉基本游戏操作，但不想在局内切出游戏查询资料的玩家。
- 希望助手只做提示，不替自己选择的玩家。

### 3.2 典型场景

- 玩家到达海克斯选择等级，面板展开，助手在三张卡片下方展示各自的联动说明和评级。
- 玩家重随其中一个海克斯，助手只刷新发生变化的卡片信息。
- 玩家暂时收起海克斯面板，助手隐藏展示内容，但不把本轮选择误判为已完成。
- ApexLOL 数据暂时不可用，助手显示“暂无数据”，不编造推荐结论。
- OCR 置信度不足，助手提示疑似结果或要求用户手动确认，不直接拿不可靠名称查询。

## 4. 核心功能

### 4.1 首次校准

本项目必须提供首次校准流程。用户在无边框模式下进入海克斯选择界面后，选择游戏所在显示器，并在截图上标记：

- 左、中、右三张海克斯名称区域。
- 左、中、右三张海克斯卡片底部锚点区域。
- 底部海克斯展示 / 隐藏按钮区域。

校准结果用于后续 OCR 裁剪、Overlay 定位和面板状态判断。用户修改分辨率、显示模式、UI 缩放、显示器布局或游戏 UI 后，产品需要提供重新校准入口。

### 4.2 局内监听

助手通过 LOL Live Client Data API 等本地允许的只读接口识别当前英雄和等级。普通监听状态下，助手不应持续截图或持续 OCR。

当玩家等级达到海克斯触发档位，并且存在未处理档位时，助手进入待识别状态。海克斯档位按 3 / 7 / 11 / 15 级处理；如果玩家一次性积累多个未处理档位，产品需要连续处理多轮选择。

### 4.3 海克斯识别

海克斯名称识别使用本地离线 OCR。产品不得依赖云 OCR 服务，不得上传游戏截图、局内画面或海克斯名称区域到第三方 OCR 平台。

OCR 只处理用户校准的三个名称区域，不做整屏无差别识别。识别结果需要经过保守匹配；低置信度、无法匹配或冲突结果不得直接当成标准海克斯名称。

### 4.4 Overlay 展示

Overlay 是透明、置顶、点击穿透的展示层，用于把说明卡片显示在三张海克斯卡片下方。Overlay 不得拦截鼠标、抢占输入焦点或遮挡选择按钮、重随按钮、海克斯名称等关键操作区域。

每张说明卡片至少展示：

- 评级。
- 当前英雄与海克斯名称。
- 简短推荐说明。
- 必要风险或适用条件。
- 数据来源 ApexLOL。
- 来源页面入口。

### 4.5 数据查询与缓存

助手根据当前英雄和识别到的海克斯名称查询 ApexLOL，并把结果展示给用户。产品必须清楚标注“数据来源：ApexLOL”。

产品可以做本地缓存，缓存粒度建议按“英雄 + 海克斯”组织。命中缓存时先展示缓存内容；未命中时请求来源页面；请求失败、解析失败或内容不可靠时显示“暂无数据”，并保留用户可理解的失败状态。

## 5. 非目标

本项目不做以下事情：

- 不做自动选择海克斯。
- 不做自动重随。
- 不做自动点击、自动确认或模拟玩家输入。
- 不做胜率预测软件或对局自动决策系统。
- 不做完整 ApexLOL 数据爬取和离线镜像。
- 不做游戏窗口改造、游戏界面替换或游戏进程插件。
- 不承诺独占全屏模式可稳定使用。
- 不以绕过反作弊、突破系统限制或提升自动化程度作为目标。

## 6. 安全边界

本项目必须继承以下边界，并作为默认运行路径的硬性约束：

- 不注入游戏进程。
- 不 Hook 游戏、客户端或系统图形链路。
- 不读取或修改游戏内存。
- 不扫描游戏内存地址或解析进程内部对象。
- 不自动点击、不自动选择、不自动重随、不自动确认。
- 不模拟键鼠输入。
- 不绕过游戏客户端、反作弊组件或系统权限限制。
- 不干扰游戏公平性。

默认运行路径只允许使用：

- LOL 本地允许的只读接口。
- 显示器级截图。
- 用户校准区域。
- 本地离线 OCR。
- 本地缓存。
- 透明置顶点击穿透 Overlay。

进程枚举、窗口标题读取、游戏目录扫描等能力不应进入默认运行链路。如未来确需诊断能力，也必须由用户显式开启，只用于环境排查，不参与识别、推荐、展示或状态判断。

## 7. 显示模式要求

本项目 MVP 默认支持无边框模式。现有验证结论显示，无边框模式更适合作为截图、OCR、Overlay 和状态机的主验收路径。

独占全屏只作为风险项：

- 不作为默认可用前提。
- 不作为 MVP 必达能力。
- 不得因为截图非黑屏就直接判定可用。
- 必须结合旧帧判断、连续刷新和人工样本确认。
- 如果风险检测失败，应提示用户切换无边框模式。

窗口模式可以作为辅助验证场景，但核心体验和验收重点应放在无边框模式。

## 8. 数据来源

### 8.1 游戏状态

当前英雄、等级和基础局内状态来自 LOL Live Client Data API 等本地允许的只读接口。助手不得通过进程扫描、窗口标题、游戏目录或内存读取判断玩家是否在局内。

### 8.2 海克斯名称

海克斯名称来自显示器级截图后的本地离线 OCR。OCR 输入只限用户校准的三张名称区域。

### 8.3 推荐内容

推荐信息来自 ApexLOL。产品必须展示来源标识，并提供打开来源页面的入口。来源请求失败或页面结构变化时，产品必须显示失败兜底状态，不能伪造结果。

### 8.4 本地数据

本地可以保存校准配置、识别日志、查询缓存、用户手动修正记录和诊断样本。保存内容应服务于用户体验和问题排查，不应保存与产品功能无关的敏感信息。

## 9. 用户流程

### 9.1 首次使用

1. 用户启动助手。
2. 助手提示推荐使用 LOL 无边框模式。
3. 用户进入一局海克斯乱斗，并等待海克斯选择界面出现。
4. 用户选择游戏所在显示器。
5. 助手获取显示器级截图。
6. 用户完成名称区域、底部锚点区域和展示 / 隐藏按钮区域校准。
7. 助手保存校准配置，并提示后续可重新校准。
8. 用户进入正常局内监听。

### 9.2 正常局内使用

1. 助手通过本地只读接口读取当前英雄和等级。
2. 玩家达到海克斯选择档位后，助手进入待处理状态。
3. 玩家展开海克斯面板。
4. 助手识别三张海克斯名称。
5. 助手查询缓存或 ApexLOL 来源。
6. Overlay 在三张卡片下方展示说明。
7. 用户自行阅读并做选择。
8. 当前轮选择结束后，助手隐藏 Overlay，并等待下一档位。

### 9.3 重随流程

1. 用户手动重随某一张海克斯。
2. 助手检测到对应位置名称变化。
3. 助手只刷新该位置 OCR、查询和展示内容。
4. 其他位置维持原有内容，除非也发生变化。

### 9.4 面板收起流程

1. 用户手动收起海克斯面板。
2. 助手检测到卡片不可见。
3. 助手隐藏 Overlay，暂停 OCR。
4. 助手不把面板收起判定为选择完成。
5. 用户再次展开面板后，助手恢复识别和展示。

## 10. 异常流程

- 未完成校准：不进入 Overlay 自动展示，提示用户先校准。
- 找不到目标显示器：提示重新选择显示器，不继续截图和 OCR。
- 截图黑屏：提示截图不可用，建议切换无边框模式。
- 疑似旧帧：停止自动识别，提示重新截图或切换显示模式。
- OCR 低置信度：显示疑似结果或要求手动确认，不直接查询。
- OCR 未识别：保留当前 slot 状态，连续失败后提示用户重新校准或手动输入。
- ApexLOL 请求失败：展示“暂无数据”，保留来源入口或重试入口。
- ApexLOL 页面解析失败：展示“暂无数据”，记录失败原因。
- Overlay 创建失败：不展示浮层，提示用户使用应用内列表视图或重新启动。
- Overlay 坐标异常：隐藏 Overlay，提示重新校准。
- 独占全屏不可用：提示切换无边框模式，不把该场景作为默认功能失败。
- 本地只读接口不可用：提示未检测到可读取的局内状态，保持待机，不扫描进程或读取内存。

## 11. MVP 范围

MVP 必须包含：

- 无边框模式下的首次校准。
- 显示器级截图和黑屏 / 旧帧风险提示。
- 通过本地只读接口读取当前英雄和等级。
- 3 / 7 / 11 / 15 级待处理档位。
- 本地离线 OCR 识别三张海克斯名称。
- 低置信度和未识别兜底。
- ApexLOL 查询、来源展示和失败兜底。
- 本地缓存。
- 透明置顶点击穿透 Overlay。
- 面板展开、收起、选择完成候选和重随变化的基础状态处理。
- 重新校准入口。
- 基础日志和诊断样本，支持排查截图、OCR、Overlay 和来源查询问题。

MVP 验收不包括独占全屏稳定可用。

## 12. 后续范围

后续版本可以考虑：

- 更完善的多显示器体验和显示器切换。
- 更丰富的 OCR 样本回放和人工修正流程。
- 更完整的长时间局内状态恢复。
- 不同分辨率、UI 缩放、地图和画质的兼容性扩展。
- 应用内列表视图，作为 Overlay 不可用时的兜底展示。
- 用户自定义评级显示、卡片尺寸和展示位置。
- 查询缓存管理和来源数据更新时间提示。
- 更多数据来源的对比展示，但必须保持来源透明和失败兜底。

## 13. 验收口径

本项目验收应优先覆盖无边框模式下的真实局内流程。每个关键结论都需要有可复查证据，包括截图样本、识别结果、查询状态、Overlay 可见性和点击穿透结果。

通过标准：

- 用户能完成校准并在后续局内复用。
- 海克斯面板展开时，三张说明卡片显示在对应位置下方。
- Overlay 不影响玩家点击、重随或选择。
- OCR 成功时展示匹配结果；失败时有明确兜底。
- ApexLOL 成功时展示来源内容；失败时不编造数据。
- 面板收起不被误判为选择完成。
- 重随后只刷新发生变化的位置。
- 低可疑面安全边界没有被破坏。

不通过标准：

- 使用注入、Hook、读内存、自动点击或自动选择。
- 默认依赖独占全屏。
- 依赖云 OCR 服务识别局内截图。
- Overlay 拦截游戏点击或遮挡关键操作。
- OCR 或来源请求失败时仍展示伪造结论。
- 未校准就进入自动展示。
