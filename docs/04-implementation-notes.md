# 本项目实施注意事项

## 迁移原则

本项目不应直接复制旧 POC 的页面和命令堆叠。旧 POC 中有大量诊断入口、临时脚本、旧 OCR 基线和人工验证辅助代码，适合当作证据库，不适合作为产品代码骨架。

迁移时只保留已经验证且边界清楚的能力：

- 显示器级截图和黑屏 / 旧帧诊断思路。
- 用户校准区域、归一化坐标和离线样本回放。
- PP-OCRv4 rec ONNX 模型经 Rust `ort` 推理的 OCR 路线。
- Live Client Data API 读取英雄和等级。
- 透明、置顶、点击穿透 Overlay 的实现经验。
- ApexLOL 查询、缓存和失败兜底原则。
- 测试日志、截图样本和回归报告格式。

## 建议本项目启动顺序

1. 建立空 Tauri + Rust + React 项目，先完成运行、构建、打包骨架。
2. 接入配置和日志目录，先把测试留痕能力做成稳定基础设施。
3. 实现截图和校准，不接 OCR、不接 Overlay。
4. 接入 OCR 引擎和离线回归命令，先通过样本集验收。
5. 接入 Live Client API 和状态机，只做状态展示。
6. 接入 Overlay，仅展示静态测试卡片并验收点击穿透。
7. 打通 OCR -> ApexLOL -> Overlay，但仍保留手动触发和日志。
8. 最后再做低频自动监听、重随刷新和完整局内闭环。

## 代码组织建议

本项目应避免把所有 Tauri command 都堆在一个文件里。建议按职责拆分：

- `capture`：显示器枚举、截图、黑屏和旧帧检测。
- `calibration`：校准配置、归一化坐标、样本读取。
- `ocr`：裁剪、预处理、ONNX 推理、词库匹配、调试图。
- `live_client`：本地只读接口状态、英雄和等级。
- `state_machine`：海克斯触发档位、面板状态、重随变化、异常恢复。
- `overlay`：窗口创建、定位、点击穿透、展示数据同步。
- `apex`：ApexLOL 查询、解析、缓存、失败兜底。
- `telemetry`：测试日志、样本索引、回归报告。

## OCR 注意事项

OCR 只处理用户校准后的三块名称区域，不能对整屏做无差别 OCR。正式链路应输出：

- slot 编号。
- 裁剪区域。
- OCR 原始文本。
- 置信度。
- 标准化名称。
- 匹配分。
- 引擎名称和耗时。
- 调试图路径。

低置信度和低匹配分必须进入疑似结果或人工修正流程，不能强行产出标准名称。

## Overlay 注意事项

Overlay 必须先在真实 Windows 桌面验收：

- 可见。
- 不抢焦点。
- 点击穿透。
- 不遮挡选择按钮、重随按钮和展示 / 隐藏按钮。
- 面板收起时隐藏。
- 重新展开时恢复。

独占全屏如果不能稳定覆盖，不允许转向注入或 Hook，只能回到无边框模式或独立窗口兜底。

## 打包注意事项

PP-OCRv4 rec 模型和 ONNXRuntime 动态库必须作为明确资源处理：

- 开发环境可用 `ORT_DYLIB_PATH` 指向本机库。
- 发布包必须包含 Windows 对应的 `onnxruntime.dll`。
- 模型文件应放在应用资源目录，并在启动时记录实际加载路径。
- 如果模型或动态库缺失，界面必须提示 OCR 不可用，而不是静默失败。

## 旧 POC 中不建议迁移的内容

- 旧 Tesseract 前端 OCR 主流程。
- 多个历史 OCR 赛马 worktree。
- POC 页面里混合的诊断、校准、OCR、状态机、ApexLOL、Overlay 大页面。
- 针对单张截图调试留下的临时样本和旧报告。
- 独占全屏作为默认可用前提的任何假设。
